--------------------------------------------------------------------------------
1. ONBOARDING & ACCOUNT MODEL (INITIAL IDEA)
--------------------------------------------------------------------------------

- User provides username + password during onboarding.
- Hash the password; store username + password_hash in Supabase.
- Give each account a unique ID; use that ID as the foreign key everywhere else for efficient querying.

Notes:
- Stable user ID as PK/FK across tables is the right shape: one canonical user row, foreign keys everywhere else.
- Hashing passwords yourself is doable but you own: salt rounds, rotation, reset flows, rate limiting, and not leaking whether a username exists.

Two paths:
  A) DIY table — e.g. accounts(id uuid PK, username unique, password_hash, created_at). App verifies hash on login; you manage sessions (JWT/cookies) yourself.
  B) Supabase Auth — UUID in auth.users, password hashing handled for you, reset flows, RLS tied to auth.uid(). Often add a profiles table keyed by auth.users.id with username unique.

For querying efficiency, both give a UUID primary key; difference is mostly security and maintenance, not query shape.

If custom username/password table:
- Use a strong slow hash (Argon2id or bcrypt) with per-user salt (libraries handle this).
- Rate-limit login attempts.
- Decide session strategy early (Supabase client + JWT vs custom tokens) — affects how RLS references “current user.”


--------------------------------------------------------------------------------
2. SCHEMA ANCHOR — user_id EVERYWHERE
--------------------------------------------------------------------------------

- Every row that belongs to a person: user_id uuid NOT NULL, FK to canonical user (auth.users or your accounts table).
- Index (user_id, …) on tables filtered by user + date or user + id.


--------------------------------------------------------------------------------
3. IDENTITY VS APP PROFILE
--------------------------------------------------------------------------------

Identity (Auth): login — email/username, password, sessions (often Supabase Auth).

App profile (product): display name, training frequency, activity level, calorie goal, weight goal mode, timezone — maps to Settings + onboarding today.

Store app fields in profiles (or user_settings), one row per user, keyed by user_id.

Keeps login separate from “how Vinland knows your calorie goal.”


--------------------------------------------------------------------------------
4. VINLAND DOMAIN → TABLES (HIGH LEVEL)
--------------------------------------------------------------------------------

Vinland today: fitness + nutrition journaling, workout library, week scheduling, daily weight, goals in Settings, local AsyncStorage only.

A) Workout library
   - workout_templates — one row per saved workout; blocks/exercises as child rows or JSON (warm up / main / cool down, weighted/cardio/circuit).

B) Week / calendar
   - scheduled_workouts or day_plans — user_id, local_date (or date), workout_template_id nullable, ordering; optional snapshot if past days must stay immutable.

C) Daily journal / completion
   - Home + Stats: checks, timer start/end, cheat meal, etc.
   - Option 1: journal_days — one row per user per calendar day, JSON for task state.
   - Option 2: normalized journal_tasks for SQL-friendly aggregates.
   - Stats (streaks, completion) → aggregates from rows or materialized summaries later.

D) Weight
   - weight_entries — user_id, date, value + unit, locked after save; unique (user_id, date).

E) Nutrition
   - Calorie goal may live on profile; daily state (cheat meal, etc.) in nutrition_days or folded into journal_days.

Prefer one clear “source of truth per calendar day” if possible: either one journal_days row with structured JSON or a few small tables keyed (user_id, date).


--------------------------------------------------------------------------------
5. ENTITY IDS (MENTAL MODEL)
--------------------------------------------------------------------------------

- Template ID — stable library entity.
- Scheduled / journal instance — tied to a date; may reference template_id for lineage (“what was this day based on?”).


--------------------------------------------------------------------------------
6. SYNC / OFFLINE (DECIDE EARLY)
--------------------------------------------------------------------------------

App is local-first today. With Supabase:

- Simple: updated_at per row, last-write-wins; optional source_device_id later.
- v1 “online after login” is simpler than offline queue.

Harder: full offline + conflict resolution — phase 2.


--------------------------------------------------------------------------------
7. ROW LEVEL SECURITY (SUPABASE)
--------------------------------------------------------------------------------

If using Supabase Auth: policies like SELECT/INSERT/UPDATE/DELETE WHERE user_id = auth.uid() on all user-owned tables.

Service role bypasses RLS — server/admin only, not client.


--------------------------------------------------------------------------------
8. ONBOARDING IN THE CLOUD
--------------------------------------------------------------------------------

Flow: sign up → Auth creates user → profile row (trigger on signup or first open) → onboarding writes profile fields + defaults.

Client can keep FirstLaunchOnboarding UX; server only enforces “complete profile” if you need that for certain features later.


--------------------------------------------------------------------------------
9. WHAT TO KEEP SIMPLE AT FIRST
--------------------------------------------------------------------------------

Reasonable v1:
- profiles — wide columns matching Settings
- workout_templates — possibly JSON for exercise structure if it matches current SavedWorkout shape
- journal_days — one row per user per date, JSON for tasks/timer/nutrition flags

Then split out weight_entries early if weight graphs need indexed time series.

Add scheduled_workouts as its own table when week queries need to be fast without scanning large JSON.


--------------------------------------------------------------------------------
10. OPEN DECISIONS (NAIL WHEN IMPLEMENTING)
--------------------------------------------------------------------------------

1) Supabase Auth vs custom username/password table — touches every RLS policy and login flow.
2) One row per day vs many normalized tables — trade SQL analytics vs ease of migration from current JSON journal.
3) Web + mobile same account — session strategy; anonymous trial vs account required from day one.
4) Migrating existing AsyncStorage — one-time “upload local backup” vs start fresh in cloud.
