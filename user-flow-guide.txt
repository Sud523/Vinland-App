================================================================================
VINLAND — USER FLOW GUIDE (Supabase-backed)
================================================================================
This guide walks through how a user moves through the app and what happens for
each action. Persistent state is stored in Supabase (Postgres) via the client;
legacy AsyncStorage is used for one-time automatic migration (when
profiles.local_data_migrated_at is still null) and optionally from Settings via
“Import from this device” when migration was skipped on this phone (e.g. another
device migrated first).

Legend:
  READ  = load from Supabase (or in-memory React state)
  WRITE = save to Supabase (or profile patch)
  STATE = React in-memory state only (until a WRITE)
  AUTH  = Supabase Auth session (email/password)

--------------------------------------------------------------------------------
1. APP LAUNCH
--------------------------------------------------------------------------------
• App loads App.tsx → AuthProvider → (when session exists) UserPrefsProvider →
  LocalDataMigrationGate → GestureHandlerRootView, SafeAreaProvider,
  NavigationContainer, Main tabs, FirstLaunchOnboarding overlay, StatusBar.

• While AuthProvider resolves the session: loading spinner.

• If no AUTH session → AuthScreen only (email / password sign-in or sign-up).

• After AUTH:
    UserPrefsProvider READ `profiles` row for `auth.uid()` (display name, prefs,
    onboarding_step / onboarding_complete, local_data_migrated_at, etc.).

• LocalDataMigrationGate:
    - If `profiles.local_data_migrated_at` is null → optional READ legacy
      AsyncStorage keys; if any data exists, WRITE journal + templates + merged
      profile fields to Supabase, then WRITE `local_data_migrated_at`, clear
      legacy keys.
    - Gate shows a brief spinner until this check finishes so tabs never READ
      an empty journal while migration is still uploading.

    - If `local_data_migrated_at` is already set (another device ran migration
      first), this device skips automatic upload of legacy keys; the user can
      still merge phone-only library/journal from Settings → “Import from this
      device” (see §8).

• If onboarding_complete is false → FirstLaunchOnboarding modal (see §2).

• Tabs render; each tab screen runs its own READ on focus (journal / library).

--------------------------------------------------------------------------------
2. FIRST-LAUNCH ONBOARDING (modal)
--------------------------------------------------------------------------------
Steps are persisted with `profiles.onboarding_step` (WRITE via UserPrefsContext).

Typical flow:
  a) Name → setDisplayName (WRITE profiles.display_name) + local state
  b) Workouts per week → setWorkoutsPerWeek (WRITE profiles.workouts_per_week)
  c) Activity level → setActivityLevel (WRITE profiles.activity_level)
  d) Daily calories → setDailyCalorieGoal (WRITE profiles.daily_calorie_goal)
  e) Optional: Cutting / Bulking:
       - READ loadData (journal), loadWeightGoal (profiles-backed)
       - commitWeightGoalForMode → WRITE profiles weight_goal_* columns

Finish → clearOnboardingStep (WRITE onboarding_step 0), setOnboardingComplete(true)
(WRITE onboarding_complete), user lands on main app.

--------------------------------------------------------------------------------
3. HOME TAB
--------------------------------------------------------------------------------
On focus (useFocusEffect):
  READ loadData → Supabase journal_days + journal_tasks for user
  Ensure today row exists (may WRITE saveData if append needed)
  clearStaleWorkoutInProgress → may WRITE saveData
  READ loadCheatMealState → profiles cheat_meal_* columns (week rollover may WRITE)

User actions (same product rules as before; persistence is Supabase):

3h) Rest day — No writes from Home; configured on Week.

3a–g) Start/end workout, toggles, calorie goal, cheat meal, weight — WRITE
saveData or profiles as appropriate.

--------------------------------------------------------------------------------
4. WEEK TAB
--------------------------------------------------------------------------------
On focus: READ loadData, loadSavedWorkouts

Same scheduling / rest-day / swipe-delete behavior as before; all mutations
WRITE saveData (full journal snapshot pattern).

--------------------------------------------------------------------------------
5. WORKOUTS TAB (stack)
--------------------------------------------------------------------------------
5a) List screen — READ loadSavedWorkouts on focus.

5b) Form screen — Save/delete READ/WRITE saveSavedWorkouts (replaces entire
library for the user in Supabase).

--------------------------------------------------------------------------------
6. TIMER TAB
--------------------------------------------------------------------------------
On focus: READ loadData for today’s timed tasks. Does not WRITE journal.

--------------------------------------------------------------------------------
7. STATS TAB
--------------------------------------------------------------------------------
On focus: READ loadData + loadWeightGoal (profiles). Pure compute; no WRITE.

--------------------------------------------------------------------------------
8. SETTINGS (root stack)
--------------------------------------------------------------------------------
On focus: READ loadData + loadWeightGoal

• Display name / prefs → UserPrefs setters → WRITE profiles

• Cut / Bulk → commitWeightGoalForMode → WRITE profiles weight_goal_* 

• Import from this device — READ legacy AsyncStorage snapshots
      (`loadLocalSavedWorkoutsForMigration`, `loadLocalJournalForMigration`),
      READ cloud library + full journal, merge (append workout templates whose
      `id` is not already in Supabase; append journal days whose calendar `date`
      has no row in the cloud — never overwrite same id / same date), then
      WRITE `replaceSavedWorkouts` / `replaceFullJournal` only when there are
      additions. Shows an alert with counts or “nothing to import”. Implemented
      in `utils/importLocalDeviceData.ts`.

• Sign out → supabase.auth.signOut → AUTH cleared → AuthScreen

--------------------------------------------------------------------------------
9. STREAKS AND STATS RULES
--------------------------------------------------------------------------------
Unchanged — see utils/stats.ts and utils/workouts.ts (in-memory Day[]).

--------------------------------------------------------------------------------
10. SUMMARY — MAIN SUPABASE TOUCHPOINTS
--------------------------------------------------------------------------------
profiles                  Settings, UserPrefsContext, cheat meal, weight goal,
                          onboarding fields, local_data_migrated_at

journal_days / journal_tasks   Home, Week, onboarding (weight goal path),
                               Stats, Timer (read-only); Settings “Import from
                               this device” may WRITE merged rows

workout_templates (+ sections + exercises)   Workouts list/form, Week scheduling;
                               Settings import may WRITE merged templates
